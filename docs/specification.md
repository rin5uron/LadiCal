# LadiCal 仕様書

## 1. 概要

LadiCal は、完全ローカルで動作する生理管理・体調記録アプリである。

- 生理周期の記録と予測を行う
- 周期に関連する体調を記録する
- 絵文字・トグル・メモで柔軟に入力できる
- 将来的には、性別を問わず使える体調ログや個人ログにも広げられる設計とする

コンセプトは「軽い・速い・消えない・見られない」。

制作背景:

- 既存の Android 版 LadiCal に近い使い勝手を iOS で継続したい
- サポート終了後も長く使える記録基盤が必要
- データ消失リスクを下げたい

## 2. 基本設計方針

- オフライン完結を最優先とする
- UI は最小構成とする
- 記録は簡単に、振り返りはしやすくする
- ユーザーごとに使い方が変わる設計にする

## 3. 画面構成

1. カレンダー（メイン）
2. 日付編集画面
3. 設定

補助機能として、必要に応じて以下を配置する。

- 保存済みリスト画面
- バックアップ / エクスポート画面

## 4. カレンダー画面

### 4.1 レイアウト

- 上部にコンパクトな月カレンダーを配置する
- 下部に選択日詳細エリアを配置する
- カレンダー画面は常時表示を基本とする

役割は以下の通り。

- カレンダー: 一覧と日付選択
- 詳細エリア: 情報表示

### 4.2 日付セル表示ルール

各日付セルには以下を表示する。

- 絵文字: 最大 2〜3 個
- 生理バー: 赤
- 排卵バー: 青
- メモ有無インジケータ: 点
- 画像有無インジケータ: 検討中

テキストは表示しない。

### 4.3 詳細エリア

選択した日付について、以下を表示する。

- 日付
- 絵文字（大きく表示）
- テキスト
- 画像サムネイル

この詳細エリアを、1 日ごとのデータ最小表示単位とする。

長文メモは「続きを読む」から入力画面へ遷移する。

## 5. 入力画面

### 5.1 入力構成

1. 絵文字選択
2. カスタム項目入力
3. テキスト
4. 画像

### 5.2 絵文字入力

- 初期絵文字は約 10 個
- 横スクロールまたはグリッド表示
- タップで即選択
- 複数選択可能

主目的は、最速入力と直感操作。

### 5.3 カスタム項目

- ユーザー定義項目を作成できる
- 項目名は初期実装では「項目1」「項目2」のような形式を想定する
- 各項目は表示 / 使用をトグルで制御できる
- 並び替え可能にする
- 各項目に画像を設定できる設計を想定する

例:

- 生理
- 頭痛
- 手荒れ

無効化した項目の既存データをどう扱うかは未確定だが、初期方針は「新規入力では非表示、既存データは保持」。

### 5.4 テキスト

- 長文対応
- URL を自動リンク化

用途:

- 日記
- 詳細記録
- 自由ログ

### 5.5 画像

- 1 日最大 1〜3 枚
- ローカル保存
- サムネイル表示
- 画像 / リスト / 保存機能は MVP に含める

## 6. カスタム項目

ユーザーが自由に項目を作成できる。

種類:

- toggle
- emoji
- number（将来対応）

方針は「すべて同一構造で扱う」こと。

絵文字カテゴリは、デフォルトに加えて追加枠を持てる設計を検討する。

## 7. リスト機能

### 7.1 概要

- メモを保存して一覧化する
- 設定で ON / OFF を切り替えられる

### 7.2 保存内容

- 日付
- メモ
- URL（任意）

### 7.3 用途

- 記録の蓄積
- 振り返り

## 8. 基礎体温

- 数値入力
- 時系列グラフ表示

ただし、設定項目と UI が重くなりやすいため MVP からは外す方向を第一候補とする。

## 9. データ管理

初期方針は以下の通り。

1. ローカル保存
2. 毎日 12 時の自動バックアップ
3. CSV エクスポート
4. iCloud バックアップ
5. Google Drive バックアップ（将来）

要件:

- 完全オフライン動作
- データ消失防止
- 保存基盤は Core Data とする

## 10. セキュリティ

- パスコードロック
- Face ID
- リストタブ非表示設定

テーマ切り替えやパスコードなどの一般設定は設定画面に集約する。

## 11. 設定画面

初期設計では、以下の項目を持つ。

- 生理日一覧
- 周期 / 現在何日目 / 平均周期 / 平均生理期間
- 一般設定（テーマ / パスコード）
- ヘルスケアデータ連携
- 体調管理設定（周期など）
- 妊娠モード / 妊娠設定
- 入力項目設定
- カレンダー表示設定
- 祝日編集（優先度低）
- データ管理（バックアップ / メール送付など）
- アプリ情報 / お知らせ / 問い合わせ / バージョン表示

データ管理と一部設定は、初期実装では簡素化前提で再設計する。

## 12. UI / UX ルール

- 初期は最小構成にする
- 入力は 1〜2 タップで完結できることを目標にする
- 情報表示の優先順位は以下とする

1. 絵文字
2. テキスト（短文）
3. 画像
4. 補助情報（将来: 体温など）

- テーマは白 / 黒のみ
- 装飾は極力排し、軽量性を優先する

## 13. データモデル（初期）

### Record

- date
- note
- imageCount
- hasListEntry

### CustomItem

- id
- name
- type（toggle / emoji / number）
- isEnabled
- sortOrder
- iconImagePath（仮）

### CustomValue

- recordId
- itemId
- value

### SavedListItem

- id
- date
- note
- url

レコードとカスタム項目の責務分割は、実装前に追加整理が必要。

## 16. 実装フロー

初心者向けの実装順は以下を基本とする。

1. Xcode プロジェクトを作る
2. Core Data モデルを定義する
3. カレンダー画面を作る
4. 日付編集画面を作る
5. 画像保存をつなぐ
6. リスト機能を作る
7. 設定画面を作る
8. CSV とバックアップを作る
9. 実機で使いながら調整する

補足:

- iPhone では「毎日12時に必ず実行」の保証が難しい
- そのため自動バックアップは「毎日1回を目指す」前提で設計する

## 17. issue 運用

このリポジトリでは、ドキュメントを増やしすぎず issue 駆動で進める。

運用方針:

- 全体仕様の正本はこの `specification.md`
- データ保存の正本は `data-model.md`
- 実装作業は issue で分ける
- issue で決まった仕様は、必要に応じてこの仕様書かデータモデルへ戻す

issue の詳細ルールは `docs/issue-rules.md` を使う。

## 18. 設計の核心

- トグルは残す
- 絵文字を主役にする
- カレンダーは最小表示にする
- 詳細エリアで情報を補完する

## 19. 最終ゴール

ユーザーごとに使い方が変わる、長く使われるパーソナルログツールを目指す。

## 20. 実装前メモ

実装開始時には、以下を先に決める。

- Record と CustomItem の責務分離
- Core Data エンティティ定義
- 画像保存方式
- カレンダーの月表示ロジック
- 入力画面の遷移方式
- セキュリティ機能の適用範囲
- 自動バックアップの実行方式
- CSV エクスポート対象項目
