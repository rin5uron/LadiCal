# LadiCal issue 運用ルール

## 1. このドキュメントの目的

このプロジェクトでは、ドキュメントを増やしすぎず、issue 駆動で実装を進める。

そのため、issue は「何をやるか」「何が決まったか」を後から追える形で作る。

## 2. 基本方針

- 仕様の正本は `docs/specification.md` と `docs/data-model.md`
- issue は実装タスク、検討、バグ、方針決定を管理する場所
- issue で仕様が確定したら、必要に応じて正本ドキュメントへ反映する
- issue だけに重要仕様を閉じ込めない

## 3. 作成方法

- issue は基本的に CLI で作成する
- `gh issue create` を使う
- Web で作ってもよいが、タイトルと本文の形式はこのルールに合わせる

例:

```bash
gh issue create --title "【課題】カレンダー画面の土台を作る" --body "（本文）" --label task
```

## 4. 種別

タイトルの先頭に種別を入れる。

- `【課題】`
- `【方針】`
- `【エラー】`
- `【ログ】`
- `【学び】`

このプロジェクトでは、まず以下を中心に使う。

- `【課題】`: 実装タスク
- `【方針】`: 仕様や運用の決定
- `【エラー】`: バグや詰まり

## 5. タイトルルール

- 先頭に種別を入れる
- できるだけ短く書く
- 日付はタイトルに入れない
- 何をする issue かが一読で分かる言葉にする

良い例:

- `【課題】Core Data の Record エンティティを定義する`
- `【課題】日付編集画面の UI を作る`
- `【方針】画像はアプリ内にコピー保存する`

## 6. ラベル

ラベルは英字で 1 つ付ける。

| 種別 | ラベル |
|------|--------|
| 【課題】 | task |
| 【方針】 | policy |
| 【エラー】 | bug |
| 【ログ】 | log |
| 【学び】 | study |

## 7. 本文ルール

本文には最低限、次を書く。

- 背景
- やること / 決めたいこと
- 完了条件

タスク issue の例:

```md
## 背景
カレンダー画面がまだ存在しない。

## やること
- 月表示の土台を作る
- 日付タップを実装する
- 下部詳細エリアを仮表示する

## 完了条件
- 月カレンダーが表示される
- 日付選択で詳細が切り替わる
```

## 8. 更新ルール

- 「上書きして」と言われたら、本文を更新する
- 進捗メモだけをコメントに積みすぎない
- 最終的な決定事項は本文に残す

## 9. issue の分け方

1 issue = 1 テーマを基本とする。

分け方の例:

- カレンダー画面
- 日付編集画面
- Core Data モデル
- 画像保存
- リスト機能
- 設定画面
- CSV / バックアップ

大きすぎる issue は分割する。

## 10. このプロジェクトでのおすすめ運用

- `specification.md` を読み、機能を 1 つ選ぶ
- その機能を `【課題】` issue にする
- 実装中に設計判断が必要なら `【方針】` issue を切る
- 決定したら仕様書かデータモデルに戻す

この流れで、仕様書と issue が分裂しにくくなる。
